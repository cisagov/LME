# LME Remote Elasticsearch Deployment - Technical Specification

## Overview

This document outlines the technical approach for extending LME's Ansible automation to support remote Elasticsearch deployment, with considerations for future cluster operations, backup strategies, and upgrade paths.

### Goals

1. Deploy Elasticsearch to remote servers via Ansible (starting point)
2. Support both local and remote installation from the same codebase
3. Enable future addition of other services (Kibana, Wazuh, Fleet) to remote nodes
4. Establish proper backup and recovery procedures for distributed deployments

---

## Current Architecture Analysis

### Existing Playbook Structure

The current `site.yml` is hardcoded for local installation:

```yaml
- name: Setup LME
  hosts: localhost
  connection: local
  become: no
```

### Existing Role Structure

The codebase already has a modular role-based structure:

| Role | Purpose |
|------|---------|
| `base` | OS setup, directories, password generation |
| `nix` | Nix package manager installation |
| `podman` | Container runtime, secrets, quadlets |
| `elasticsearch` | ES configuration and user setup |
| `kibana` | Kibana deployment |
| `wazuh` | Wazuh manager deployment |
| `fleet` | Fleet server deployment |
| `dashboards` | Dashboard import |
| `cleanup` | Post-install cleanup |

### Secrets Management

Current implementation uses Podman's shell secret driver:
- Vault password stored in `/etc/lme/pass.sh`
- Encrypted secrets in `/etc/lme/vault/`
- Quadlets reference secrets via `Secret=elastic,type=env,target=ELASTIC_PASSWORD`

The ES quadlet requires these secrets:
- `Secret=elastic,type=env,target=ELASTIC_PASSWORD` - Bootstrap password (first node only)
- `Secret=kibana_system,type=env,target=ELASTICSEARCH_PASSWORD` - For Kibana connection
- `Secret=kibana_system,type=env,target=KIBANA_PASSWORD` - Kibana system password

### Certificate Generation

Certificates are generated by the `lme-setup-certs` container using `elasticsearch-certutil`:
- CA and service certificates created from `config/setup/instances.yml`
- Stored in the `lme_certs` Podman volume
- Currently only includes localhost IPs

The `instances.yml` file defines DNS names and IPs for certificate generation:
- elasticsearch, kibana, fleet-server, wazuh-manager, etc.
- All currently set to localhost/127.0.0.1

### ES Quadlet Configuration

The current `lme-elasticsearch.container` uses:
- `discovery.type=single-node` (no clustering)
- TODO comment indicates need for cluster configuration
- Secrets for bootstrap and service passwords
- Local volume mounts for data and certificates

## Implementation Plan

### Phase 1: Inventory-Driven Playbook

**Objective:** Remove hardcoded local connection, use inventory to determine targets.

#### Changes to `site.yml`

Remove hardcoded connection and host settings, allow inventory to control targeting:

```yaml
- name: Setup LME
  hosts: "{{ target_hosts | default('localhost') }}"
  become: yes
  # Remove: connection: local (let inventory decide)
```

#### New Inventory Files

Create `ansible/inventory/` directory with example inventories. For Ansible inventory syntax, see [[6]](#references):

**`local.yml`** - For single-server local installation:
```yaml
all:
  hosts:
    localhost:
      ansible_connection: local
```

**`remote.yml`** - Template for remote deployment:
```yaml
all:
  children:
    elasticsearch:
      hosts:
        es1:
          ansible_host: 10.0.0.5
          ansible_user: admin
    # Future expansion:
    # kibana:
    #   hosts:
    #     kibana1:
    #       ansible_host: 10.0.0.6
```

### Phase 2: Certificate Generation for Remote Nodes

**Objective:** Generate certificates on control node with correct hostnames/IPs, push to remotes.

#### Option A: Centralized Generation (Recommended)

1. Create new `certs` role that runs on control node
2. Template `instances.yml` to include remote node IPs/hostnames
3. Generate all certificates locally using existing cert generation mechanism
4. Push certificates to remote nodes before service startup

#### Template for `instances.yml`

The certificate generation needs the actual IPs/hostnames that will be used for TLS connections. These are typically the same as `ansible_host` (the SSH target), but could be different if ES uses different network interfaces.

```yaml
instances:
  - name: "elasticsearch"
    dns:
      - "lme-elasticsearch"
      - "{{ inventory_hostname }}"
      - "localhost"
    ip:
      # Use ansible_host (SSH IP) or es_cluster_ip if defined separately
      - "{{ hostvars[inventory_hostname]['es_cluster_ip'] | default(hostvars[inventory_hostname]['ansible_host']) }}"
      - "127.0.0.1"
```

**Note:** `ansible_host` is the IP Ansible uses for SSH [[6]](#references). For certificates, we typically use the same IP unless nodes have separate cluster communication interfaces. If all nodes use the same IP for SSH and cluster communication, `ansible_host` is sufficient.

#### New Role: `ansible/roles/certs/`

This role will:
- Run certificate generation on control node (delegate_to: localhost)
- Template instances.yml with inventory variables
- Execute cert generation (via container or direct openssl/elasticsearch-certutil)
- Distribute certificates to target nodes
- Ensure proper permissions on remote nodes

### Phase 3: Templated ES Quadlet for Clustering

**Objective:** Support single-node, cluster initialization, and cluster join scenarios.

#### Create `ansible/templates/lme-elasticsearch.container.j2`

Template the quadlet file to support different deployment modes. For quadlet syntax reference, see [[5]](#references):

```ini
[Unit]
Description=Elasticsearch Container Service
Requires=lme-network.service lme-setup-certs.service
After=lme-network.service lme-setup-certs.service
PartOf=lme.service

[Service]
Restart=always
Environment=ANSIBLE_VAULT_PASSWORD_FILE=/etc/lme/pass.sh
TimeoutStartSec=5400

[Container]
ContainerName=lme-elasticsearch
Environment=node.name={{ inventory_hostname }}
Environment=cluster.name={{ es_cluster_name | default('LME') }}

{% if es_discovery_type == 'single-node' %}
# Standalone mode
Environment=discovery.type=single-node
{% else %}
# Cluster mode
# discovery.seed_hosts: IPs/hostnames:ports for finding other nodes (typically port 9300)
Environment=discovery.seed_hosts={{ es_seed_hosts | join(',') }}
{% if es_is_initial_master | default(false) %}
# cluster.initial_master_nodes: Node names (node.name values) for initial cluster formation
Environment=cluster.initial_master_nodes={{ es_initial_masters | join(',') }}
{% endif %}
{% endif %}

# ... rest of configuration (TLS, security, etc.)
Secret=elastic,type=env,target=ELASTIC_PASSWORD
Secret=kibana_system,type=env,target=ELASTICSEARCH_PASSWORD
Secret=kibana_system,type=env,target=KIBANA_PASSWORD
```

#### Inventory Variables for Clustering

**Important distinction:**
- `ansible_host` - IP/hostname Ansible uses for SSH connection (Ansible-specific)
- `es_seed_hosts` - IPs/hostnames Elasticsearch uses for cluster discovery (ES-specific)
- `es_initial_masters` - Node names (matching `node.name`) for initial cluster formation (ES-specific)

These may be the same values, but serve different purposes. For simplicity, you can use `ansible_host` values for ES cluster configuration, or define separate `es_cluster_ip` variables if cluster communication uses different addresses.

**Example Inventory:**

```yaml
elasticsearch:
  vars:
    es_cluster_name: LME
    # Option 1: Hardcode seed hosts (simple but less flexible)
    es_seed_hosts:
      - 10.0.0.5:9300  # IP:port for transport protocol
      - 10.0.0.6:9300
    # Option 2: Derive from inventory (better)
    # es_seed_hosts: "{{ groups['elasticsearch'] | map('extract', hostvars, 'ansible_host') | map('regex_replace', '$', ':9300') | list }}"
    
    # es_initial_masters must match node.name values (inventory hostnames, can be ip)
    es_initial_masters:
      - es1  # Matches node.name={{ inventory_hostname }} for es1
      - es2  # Matches node.name={{ inventory_hostname }} for es2
  hosts:
    es1:
      ansible_host: 10.0.0.5  # Ansible SSH target
      es_discovery_type: single-node  # First node / standalone
      es_is_initial_master: true
    es2:
      ansible_host: 10.0.0.6  # Ansible SSH target
      es_discovery_type: join  # Joins existing cluster
```

**How it maps to ES configuration:**

When the quadlet template renders for `es1`, it produces:
```ini
Environment=node.name=es1
Environment=discovery.seed_hosts=10.0.0.5:9300,10.0.0.6:9300
Environment=cluster.initial_master_nodes=es1,es2
```

The `cluster.initial_master_nodes` setting uses the **node names** (`es1`, `es2`) which match the `node.name` values, not IP addresses. This is required by Elasticsearch for cluster bootstrapping [[4]](#references).

**Note:** 
- `es_seed_hosts` should contain IPs/hostnames with optional ports (default 9300) that ES nodes use to communicate [[4]](#references)
- `es_initial_masters` must exactly match the `node.name` values set in the quadlet (which uses `inventory_hostname`)
- After cluster formation, `cluster.initial_master_nodes` can be removed from configuration [[4]](#references)

### Phase 4: Secrets Distribution

**Objective:** Ensure vault password and secrets are available on remote nodes for Podman shell driver.

#### Required on Each Node

1. `/etc/lme/pass.sh` - Vault password file (mode 0700)
2. `/etc/lme/vault/` - Encrypted secret files (ansible-vault encrypted)
3. `/root/.config/containers/containers.conf` - Shell driver configuration

The podman shell driver configuration references these paths:
```ini
[secrets.opts]
list = "ls {{ user_vault_dir }}"
lookup = "ansible-vault view {{ user_vault_dir }}/$SECRET_ID | tr -d '\n'"
```

#### Distribution Approach

Add tasks to `base` role or create new `secrets_distribution` role:

```yaml
- name: Push vault password to remote nodes
  copy:
    src: /etc/lme/pass.sh
    dest: /etc/lme/pass.sh
    mode: '0700'
  when: inventory_hostname != 'localhost'
  delegate_to: localhost
  run_once: true

- name: Push encrypted secrets
  synchronize:
    src: /etc/lme/vault/
    dest: /etc/lme/vault/
  when: inventory_hostname != 'localhost'
  delegate_to: localhost
  run_once: true
```

**Backup Strategy:** The vault password (`/etc/lme/pass.sh`) and vault directory (`/etc/lme/vault/`) are critical and must be backed up on the control node. These files enable decryption of all service passwords.

### Phase 5: ES-Only Playbook

**Objective:** Allow deploying just Elasticsearch without full stack.

#### Create `ansible/elasticsearch.yml`

```yaml
---
- name: Deploy Elasticsearch
  hosts: elasticsearch

```

## Cluster Operations

### Understanding Elasticsearch Cluster Behavior

#### Data Distribution

In an ES cluster, indices are divided into shards distributed across nodes:

```
Cluster with 3 nodes, index with 5 primary shards + 1 replica:

Node 1: [P0] [P3] [R1] [R4]
Node 2: [P1] [P4] [R0] [R3]
Node 3: [P2] [R2]

P = Primary shard, R = Replica shard
```

Each node contains only a subset of the total data. Filesystem-level backups of individual nodes result in incomplete data.

#### Password and Security Index

- Built-in user passwords (elastic, kibana_system, etc.) are stored in the `.security` index
- This index replicates across all cluster nodes automatically
- Password changes via API propagate instantly to all nodes
- Bootstrap password (`ELASTIC_PASSWORD` env var) is only used on first startup before security is initialized
- For subsequent nodes joining a cluster, the `.security` index already exists and replicates to them

### Backup Strategy for Clusters

#### Current Backup Mechanism (Single Node)

The existing `backup_lme.yml` performs filesystem-level backup:
- Copies `/opt/lme/` configuration
- Copies `/etc/lme/` vault and secrets
- Copies Podman volumes including `lme_esdata01`

**This approach does NOT work for clusters** because:
1. Each node has only partial data (shards)
2. Filesystem copy may capture inconsistent state
3. Cannot restore partial data to recreate complete indices
4. Shards may be in-flight during backup, causing inconsistency

#### Recommended Cluster Backup Strategy

| Component | Single Node | Cluster |
|-----------|-------------|---------|
| Configuration | Filesystem copy | Filesystem copy (from control node) |
| Secrets/Vault | Filesystem copy | Filesystem copy (from control node) |
| Certificates | Filesystem copy | Filesystem copy (from control node) |
| ES Data | Filesystem copy of volume | **ES Snapshot API** |

#### ES Snapshot API

Elasticsearch provides a cluster-aware snapshot mechanism that captures consistent state across all shards [[2]](#references). When a snapshot is initiated, the coordinating node orchestrates the process, but **each data node writes its own shard data directly to the repository**. This means all master and data nodes must have access to the snapshot repository.

**Repository Types and Requirements:**

1. **Filesystem Repository (`type: "fs"`):**
   - All nodes must have the repository path mounted at the same location
   - For clusters, this typically requires shared storage (NFS, distributed filesystem)
   - Each node writes its shards directly to the filesystem
   - Example: NFS mount at `/usr/share/elasticsearch/backups` on all nodes

2. **S3 Repository (`type: "s3"`):**
   - All nodes need network access to the S3 bucket
   - Each node writes its shards directly to S3
   - No shared filesystem required, but all nodes need S3 credentials and network access
   - Repository is "shared" by virtue of being external (S3 bucket)

3. **URL Repository (`type: "url"`):**
   - All nodes need network access to the HTTP/HTTPS endpoint
   - Typically read-only, used for restoring from external sources
   - Not suitable for creating snapshots in a cluster

**For LME Filesystem Repository:**

The ES quadlet already mounts a backup volume:
```ini
Volume=lme_backups:/usr/share/elasticsearch/backups
```

For single-node deployments, this can be a local volume. For cluster deployments:
- Option 1: Use shared storage (NFS) mounted at the same path on all nodes
- Option 2: Use S3 repository instead (requires S3 plugin and credentials)
- Option 3: Each node backs up to local storage, but this only captures that node's shards (not recommended for cluster-wide backups)

**Note:** While the coordinating node initiates the snapshot, each data node must be able to write to the repository location. For filesystem repositories in a cluster, this typically means shared storage, but S3 or other external repositories can be used if all nodes have network access.

#### Snapshot Management Tasks (Future)

Create `ansible/snapshot_elasticsearch.yml` for snapshot operations. See [[2]](#references) for complete snapshot API documentation:

```yaml
- name: Create ES Snapshot
  hosts: elasticsearch
  tasks:
    - name: Register snapshot repository
      uri:
        url: "https://{{ inventory_hostname }}:9200/_snapshot/lme_backups"
        method: PUT
        body_format: json
        body:
          type: "fs"
          settings:
            location: "/usr/share/elasticsearch/backups"
      run_once: true
      
    - name: Create snapshot
      uri:
        url: "https://{{ inventory_hostname }}:9200/_snapshot/lme_backups/{{ snapshot_name }}"
        method: PUT
        body_format: json
        body:
          indices: "*"
          include_global_state: true
      run_once: true
```

### Upgrade Strategy for Clusters

#### Critical Constraint: No ES Downgrade

**Elasticsearch does not support version downgrade.** Once data is written by a newer version, it cannot be read by older versions. The data format migrates forward on upgrade [[1]](#references).

This means:
- Traditional "rollback" to previous ES version is not possible
- If upgrade fails, restore from snapshot to fresh cluster on old version
- Config/secrets can be rolled back, but ES data cannot

#### Rolling Upgrade Procedure

The following procedure follows Elasticsearch's recommended rolling upgrade process [[1]](#references):

1. **Pre-upgrade:**
   - Create ES snapshot
   - Verify cluster health is green
   - Document current version
   - Ensure sufficient disk space

2. **For each node (one at a time):**
   ```bash
   # Disable shard allocation
   PUT _cluster/settings
   {"persistent": {"cluster.routing.allocation.enable": "primaries"}}
   
   # Stop indexing (optional, for faster recovery)
   POST _flush/synced
   
   # Stop ES on target node
   systemctl stop lme-elasticsearch
   
   # Upgrade container image
   podman pull new-image:version
   # Update quadlet if needed
   
   # Start ES on target node
   systemctl start lme-elasticsearch
   
   # Wait for node to rejoin
   GET _cat/nodes
   
   # Re-enable allocation
   PUT _cluster/settings
   {"persistent": {"cluster.routing.allocation.enable": null}}
   
   # Wait for green health before next node
   GET _cluster/health
   ```

3. **Post-upgrade:**
   - Verify all nodes on new version
   - Create new snapshot
   - Update documentation
   - Test functionality

#### Ansible Rolling Upgrade Playbook (Future)

```yaml
- name: Rolling upgrade ES cluster
  hosts: elasticsearch
  serial: 1  # One node at a time
  tasks:
    # This may take some time, so it needs investigating
    - name: Disable shard allocation
      uri:
        url: "https://{{ es_primary_node }}:9200/_cluster/settings"
        method: PUT
        body_format: json
        body:
          persistent:
            cluster.routing.allocation.enable: "primaries"
      run_once: true
      delegate_to: localhost
      
    - name: Stop ES on this node
      systemd:
        name: lme-elasticsearch
        state: stopped
        
    - name: Update container image
      # Pull new image, update quadlet
      
    - name: Start ES on this node
      systemd:
        name: lme-elasticsearch
        state: started
        
    - name: Wait for node to rejoin cluster
      uri:
        url: "https://localhost:9200/_cluster/health"
        method: GET
        validate_certs: no
      register: health
      until: health.json.status in ['green', 'yellow']
      retries: 60
      delay: 10
    
    # Also will want to check this one out as well
    - name: Re-enable shard allocation
      uri:
        url: "https://{{ es_primary_node }}:9200/_cluster/settings"
        method: PUT
        body_format: json
        body:
          persistent:
            cluster.routing.allocation.enable: null
      run_once: true
      delegate_to: localhost
```

### Disaster Recovery

#### Scenario: Failed Upgrade

If an upgrade fails mid-way:

1. **Do NOT attempt to downgrade ES** - it will not work
2. Options:
   - Complete the upgrade on remaining nodes if possible
   - Restore from pre-upgrade ES snapshot to a fresh cluster on old version
   - Restore snapshot to new cluster on new version (if data format compatible)

#### Scenario: Data Loss/Corruption

##### These will require documentation as well as an easy way to deploy a new instance 
1. Deploy fresh ES cluster with same version as snapshot
2. Restore from ES snapshot
3. Restore config/secrets from Ansible backup
4. Re-enroll agents if necessary

#### Scenario: Single Node Failure (in cluster)

1. ES automatically redistributes shards to remaining nodes (if replicas configured)
2. Replace failed node
3. New node joins cluster and receives shard copies automatically
4. Ensure proper certificate and configuration on replacement node

## Challenges and Solutions

### Challenge 1: Sharded Data Backup

**Problem:** Filesystem backup of individual nodes captures incomplete data. Each node only has a subset of shards.

**Solution:** 
- Use ES Snapshot API for data backup (cluster-aware, consistent)
- Require shared storage (NFS/S3) for cluster deployments
- Keep filesystem backup for config/secrets only
- Document that filesystem backup is insufficient for cluster data

### Challenge 2: No Version Rollback

**Problem:** ES cannot downgrade; data format changes are one-way.

**Solution:**
- Always snapshot before upgrades
- Test upgrades in staging environment first
- Plan forward-only upgrade path
- For rollback: restore snapshot to fresh cluster on old version
- Update documentation to clarify "rollback" means config rollback, not ES version rollback

### Challenge 3: Certificate Distribution

**Problem:** Each node needs certificates with correct hostnames/IPs for TLS.

**Solution:**
- Generate all certificates on control node
- Template `instances.yml` with inventory variables
- Push certificates before service startup
- Store CA on control node for future node additions
- Ensure certificates include both DNS names and IPs for each service

### Challenge 4: Secrets on Remote Nodes

**Problem:** Podman shell driver requires vault password and encrypted files locally on each node.

**Solution:**
- Push `/etc/lme/pass.sh` and `/etc/lme/vault/` to each node
- Ensure consistent secrets across all nodes
- Central management on control node, distributed via Ansible
- Backup vault files on control node (critical for recovery)

Note: The passwords may need to be reencrypted for each node

### Challenge 5: Repository Access Requirement

**Problem:** ES snapshots require all nodes to have access to the repository because each node writes its own shard data.

**Solutions:**
1. **Filesystem Repository with Shared Storage:** Mount shared NFS/distributed filesystem on all ES nodes at `/usr/share/elasticsearch/backups`
2. **S3 Repository:** Use S3-compatible repository (S3, MinIO, etc.) - all nodes need network access and credentials
3. **Azure Blob/GCS:** Use respective repository plugins - all nodes need network access

**Important:** Each data node writes its own shards directly to the repository. The coordinating node orchestrates but doesn't pull data from other nodes. Therefore:
- For filesystem repositories: Shared storage (NFS) is typically required
- For external repositories (S3, etc.): All nodes need network access, but no shared filesystem needed

**Mitigation if no shared storage:**
- Use S3 or other external repository type (requires plugin installation)
- Accept limitation: no cluster-wide snapshots with local filesystem repository
- Rely on replica shards for redundancy
- Document increased risk

### Challenge 6: Network Connectivity

**Problem:** ES nodes must communicate for cluster formation and data replication.

**Requirements:**
- Port 9200: HTTP API (client access)
- Port 9300: Transport (inter-node communication)
- Firewall rules must allow 9300 between all ES nodes
- Network latency should be low (<10ms recommended)

**Solution:**
- Document firewall requirements in deployment guide
- Provide example firewall rules
- Test connectivity before deployment

### Challenge 7: Split-Brain Prevention

**Problem:** Network partition could cause multiple masters, data inconsistency.

**Solution:**
- Configure `discovery.seed_hosts` correctly
- Set `cluster.initial_master_nodes` only on first cluster formation
- For 3+ node clusters, ensure odd number of master-eligible nodes
- Use minimum_master_nodes setting (auto-configured in ES 7.x+)

### Challenge 8: Bootstrap Password for Second Node

**Problem:** Second node joining cluster doesn't need bootstrap password, but quadlet still references it.

**Solution:**
- Bootstrap password only used on first node
- Second+ nodes join existing cluster, `.security` index replicates automatically
- Secret still needs to exist (for health checks or initial connection), but ES doesn't use it for bootstrap
- Consider making bootstrap password optional in template for join nodes

## Migration Path

### For Existing Single-Node Installations

No changes required. Existing installations continue to work with:
```bash
ansible-playbook -i inventory/local.yml site.yml
```

The local inventory maintains backward compatibility.

### For New Remote Deployments

1. Create inventory file with target hosts
2. Ensure SSH access and sudo privileges on targets
3. Run with remote inventory:
```bash
ansible-playbook -i inventory/remote.yml elasticsearch.yml -K
```

The `-K` flag prompts for sudo password.

### For Cluster Deployments

1. Configure shared storage for backups (NFS recommended Make sure to lock down)
2. Create inventory with all cluster nodes
3. Deploy first node with `es_discovery_type: single-node`
4. Deploy additional nodes with `es_discovery_type: join` and `es_seed_hosts` configured
5. Configure snapshot repository via ES API
6. Establish snapshot schedule (daily recommended)
7. Test snapshot restore procedure

## Prerequisites for Remote Deployment

### On Control Node (where Ansible runs)
- Ansible 2.9+
- SSH access to target nodes
- LME repository cloned
- Access to `/etc/lme/pass.sh` and `/etc/lme/vault/` for secret distribution

### On Target Nodes
- Ubuntu 22.04/24.04, Debian 12, or RHEL 9 (Match the first node)
- SSH server running
- Python 3 installed
- Sudo privileges for deployment user
- Network connectivity to control node
- For clusters: network connectivity between all ES nodes (port 9300)
- Sufficient disk space (see LME requirements)
- Memory limits configured (vm.max_map_count for ES)

### For Cluster Backups
- **Filesystem Repository:** Shared storage (NFS recommended) mounted at same path on all ES nodes, OR
- **S3/External Repository:** Network access and credentials for all nodes to access external repository
- Minimum 2x data size for snapshot storage
- All nodes must have write access to repository location
- Proper permissions configured (filesystem) or credentials (S3/external)
